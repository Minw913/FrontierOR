## 1. Generate Feasibility Checker
### Task Statement
Generate a standard feasibility checker (python script) that verifies the feasibility of a candidate solution produced by an algorithm or heuristic method.

### Data Must Be Referenced
(1) The formal mathematical formulation extracted from the paper: 'math_model.txt' in this directory.
(2) The implementation code of the solution approach used to produce the candidate solution: 'efficient_algorithm.py' or 'gurobi_code.py' in the this directory.
(3) An example JSON specification of a test instance configuration which is extracted from the paper: 'instance/tiny_instance.json' in the this directory.
(4) An example JSON specification of a candidate solution generated by this approach: 'efficient_solution/tiny_solution.json' or 'gurobi_solution/tiny_solution.json' in the this directory.

P.S. The loaded data instance and candidate solution strictly follow the same JSON structure and field semantics as the examples provided above.

### INSTRUCTIONS
1. Treat the listed constraints in the mathematical model as Constraint (1), Constraint (2), ..., counting strictly from top to bottom in the formulation section.
2. Design a Python script named feasibility_check.py that:
- It MUST use `argparse` to read and output all external file paths from the command line to avoid hard-coding the paths:
    - `--instance_path`: path to read the JSON file containing the data instance.
    - `--solution_path`: path to read the JSON file containing the candidate solution.
    - `--result_path`: path to write the JSON file containing the feasibility result.
- The script MUST NOT hard-code, inline, or directly embed the data instance or the candidate solution provided in this prompt.
3. Explicitly checks each hard constraint one by one. Soft constraints, penalties, or heuristic rules must be ignored. For each violated constraint:
4. **Variable domain constraints are hard constraints and MUST be checked explicitly.**
   In addition to the numbered inequality/equality constraints, the DECISION VARIABLES
   section of the mathematical model declares domain restrictions such as:
   - Binary: `x ∈ {0, 1}` — verify `val == 0 or val == 1` (within tolerance).
   - Integer: `x ∈ Z` or `x ∈ Z+` — verify `abs(val - round(val)) <= tol`.
   - Non-negativity: `x >= 0` — verify `val >= -tol`.
   - Explicit bounds: `0 <= x <= K` — verify both sides.
   These checks MUST validate the actual numeric values read from the solution JSON.
   Do NOT assume any domain constraint is "auto-satisfied by the solution
   representation" — even if the JSON encodes a variable as a list of selected
   items, verify there are no duplicates (which would violate binary domain).
   Assign each group of domain constraints its own constraint index, continuing
   the numbering from the last numbered constraint in the formulation.
   - Record the corresponding constraint index (starting from 1, matching the paper's formulation order)
   - Generate a clear, human-readable violation message
   - Compute a normalized violation magnitude for each violated constraint instance, as specified below.

### NORMALIZED VIOLATION MAGNITUDE (MANDATORY)
- Use numerical tolerance `tol = 1e-5` and `eps = 1e-5`.
- For each hard constraint, compute a non-negative `violation_amount` based on
   the comparison between its left-hand side (LHS) value computed from the candidate solution and the data instance and right-hand side (RHS)
   value specified by the model, according to the constraint operator (≤, <, ≥, >, or =).
- For ≤ or < constraints, `violation_amount` measures how much LHS exceeds RHS.
- For ≥ or > constraints, `violation_amount` measures how much RHS exceeds LHS.
- For = constraints, `violation_amount` is the absolute difference between LHS and RHS.
- A constraint is considered violated if `violation_amount > tol`. Strict
   inequalities (<, >) are evaluated using the same logic, with `tol` defining
   numerical feasibility.
- For each violation, compute a normalized violation magnitude as:
      `violation_amount / max(|RHS|, eps)`.
- Record all violation instances in a list named `"violation_magnitudes"`, where
   each entry is a dictionary with fields:
   - `"constraint"`: the constraint index (integer, matching the paper's formulation order)
   - `"lhs"`: the computed left-hand side value (float)
   - `"rhs"`: the right-hand side value from the model (float)
   - `"raw_excess"`: the raw violation amount (`violation_amount`, float)
   - `"normalizer"`: the value used for normalization (`max(|RHS|, eps)`, float)
   - `"ratio"`: the normalized violation value (float)

### OUTPUT REQUIREMENTS
The JSON file must have the following structure:
   {{
      "feasible": false,
      "violated_constraints": [1, 3, 4],
      "violations": [
      "Customer 7 visited twice",
      "Capacity exceeded on route 3"
      ],
      "violation_magnitudes": [
      {{
         "constraint": 3,
         "raw_excess": 5.0, 
         "normalizer": 20.0, 
         "ratio": 0.25
      }}
      ]
   }}

    - "feasible" must be true if and only if no hard constraint is violated
    - "violated_constraints" must list the unique indices of violated constraints (no duplicates) as numbered in the paper, with "violations" aggregated into corresponding messages.
    - "violations" must list concise textual explanations aligned with those indices
    - "violation_magnitudes" must be an empty list if no constraint is violated

### Output Path 
'feasibility_check.py' in this directory.
