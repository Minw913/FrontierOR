## Rules

### R1. Scale-parameter naming alignment (critical for [STRICT] cap)

The CSV's `primary_param` and `largest_real_world_config` may use slightly different identifier styles than the txt's Scale parameters section (which mirrors `large_instance_11.json` and `generate_multiple_instances.py`). Always use the TXT's convention throughout the file. If the CSV says `num_requests` but the generator uses `num_requests_n`, write `num_requests_n` in BOTH the Scale parameters AND the [STRICT] cap sections. Same underlying concept → same name everywhere.

Examples of alignment you'll need to do:
- CSV `num_requests` vs. large_instance_11 `num_requests_n` → use `num_requests_n`
- CSV `num_edges` vs. large_instance_11 `graph.num_edges` → use `E (num_edges)` or just `num_edges` consistently
- CSV `num_jobs` vs. large_instance_11 `num_jobs` → use `num_jobs` (already aligned)

### R2. Current-state line of the Enforcement rule

Compare large_instance_11's values against the [STRICT] cap and pick ONE of:

(a) If large_instance_11 meets the cap on ALL axes:
   "`large_instance_11.json` is AT the cap exactly, so any scale-up requires explicit user override"

(b) If large_instance_11 is STRICTLY BELOW the cap on every axis:
   "`large_instance_11.json` (axes: <field=value, ...>) is within the cap on all axes; scale-up within the cap can proceed automatically, and moving toward the cap is the default scale-up direction"

(c) If large_instance_11 EXCEEDS the cap on any axis (e.g., paper pushed to synthetic stress beyond realistic applications):
   "`large_instance_11.json` already exceeds the cap on axis <field> (value <X>, cap <Y>) — treating this as a user-endorsed override-in-effect; all new variants must still not exceed large_instance_11 on any axis"

### R3. Scale parameters must drive variable/constraint count

The Scale parameters section should list ONLY axes that make the MIP larger when increased — i.e., determine the number of binary/continuous variables or constraints. Read `math_model.txt` to identify these. Non-scale parameters (cost values, distribution shapes, benchmark family labels) belong in the Diversity section, NOT the Scale section.

### R4. Diversity dimension selection — Tier 1/2/3

Priority-ordered fallback:

**Tier 1 (preferred) — from paper's own all_instances.json:**
- Same benchmark family, different members (e.g., different TSPLIB coord sources, different Lawrence la-instance numbers)
- Non-scale parameter axis the paper already sweeps (demand distribution, cost structure, constraint tightness)
- Problem variants the paper toggles (with/without setup, with/without TW, etc.)

**Tier 2 (fallback) — default diversity dimension for the problem class:**
- VRP / TSP → Spatial Layout (uniform / clustered / grid / mixed / realistic-city)
- Scheduling → Processing-time Distribution (uniform / correlated / bimodal / triplet / heavy-tail)
- Packing / Container Optimization → Weight Distribution (uncorrelated / weakly-corr / strongly-corr / inverse-corr)
- Facility Location → Cost Structure (low-fixed/high-var, high-fixed/low-var, uniform, spatial-decay, step-cost)
- Graph Optimization → Graph Structure (ER / BA / WS / planar / complete)
- Lot Sizing → Time/Horizon Pattern (stationary / seasonal / trend / spiky / stochastic)
- Inventory Routing → Demand Pattern (stationary / cyclical / bursty / correlated)
- Network Design → Commodity/Demand Pattern (single-commodity / multi-commodity / hub-and-spoke / realistic)
- Energy / Unit Commitment → Stress/Demand Multiplier (normal / peak / extreme / outage)
- Revenue Management / Portfolio → Cardinality / Correlation Structure
- Crew Scheduling → Shift Distribution / Skill Mix

**Tier 3 (last resort):** Constraint tightness sweep at fixed scale (capacity / budget / TW utilization = 0.3 / 0.5 / 0.7 / 0.9 / 0.99).

Only fall back to Tier 2/3 if the paper's native diversity options along ONE axis sum to fewer than 4 values. Always explain in Notes why native was insufficient.

### R5. Invariants across the 5 structures

- All 5 MUST share the Scale parameters values exactly (strict axes).
- All 5 MUST be within the [STRICT] cap on every axis.
- For every Tier-2/3 synthetic variant, provide a concrete generation recipe (algorithm name, distribution parameters, seed scheme).
- `large_instance_11` is always entry #1 (the EXISTING file); entries #2–5 describe how to generate `large_instance_21.json` through `large_instance_51.json`.

### R6. External-benchmark note

If a Tier-1 diversity option requires an external benchmark file (TSPLIB, OR-library, Solomon, Taillard, DIMACS, MIPLIB, egl, PSPLIB, etc.) that isn't in the repo, just describe the expected source in Notes — the downstream Agent (Phase 3) will fetch it using the authorization in `prompt_update_instances.txt`. Do not attempt to fetch anything yourself.
